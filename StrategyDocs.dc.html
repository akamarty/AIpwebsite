Product Requirements & Architecture

v0.1 · Phase 1 MVP scope

1. Product Requirements

Problem: a Porsche VIN tells you almost nothing unless you know how to read it. Enthusiasts, buyers, and dealers currently piece together option codes across forums, PDFs, and paid reports.

Solution: paste a VIN, get the full factory story in seconds — specs, options explained in plain English, and production context — free, fast, and built for search engines to find.

Success metric (Phase 1): organic sessions to VIN pages, decode-to-signup conversion rate, and time-to-decode (<2s).

Non-goals for Phase 1: market value, collector scoring, auction data, dealer tools — decoding and explaining, done exceptionally, first.

2. User Flow

Anonymous visitor
  → lands on Homepage (organic search or direct)
  → enters VIN → VIN Results page (full decode + AI option explanations, unlocked)
  → sees Collector Intelligence teasers (Score / Value / Auction) — locked
  → prompted: "Create free account to unlock + save this car"
  → Create Account (Google / Apple / Email)
  → returns to VIN Results, now unlocked + saved to Garage
  → Dashboard: Garage, Search History, Favorites

3. Information Architecture

4. Database Schema (Supabase / Postgres)

users            id, email, auth_provider, created_at, plan
vins             id, vin (unique), model, year, generation, engine, horsepower,
                 transmission, drive_type, exterior_color, interior_color,
                 msrp, plant, production_notes, decoded_at
vin_options      id, vin_id → vins, code, name, ai_explanation
garage_entries   id, user_id → users, vin_id → vins, nickname, saved_at
search_history   id, user_id → users (nullable), vin, searched_at
option_codes     code (unique), name, category, ai_explanation   -- SEO/reference table

-- Future (architected for, not built): market_values, auction_listings,
-- collector_scores, price_alerts — all keyed on vin_id, additive only.

5. API Architecture

POST /api/decode           { vin } → parsed spec + cached AI option explanations
GET  /api/vin/:vin          public read, powers the SEO page + OG/structured data
POST /api/garage            (auth) save vin_id to user's garage
GET  /api/garage             (auth) list saved vins
GET  /api/history            (auth) recent searches
POST /api/auth/*            Supabase Auth (Google, Apple, email/password)

Edge Function: decode-vin → validates VIN, resolves factory codes, calls
AI explanation service (cached per option code — explanations are shared
across every car with that option, so this is O(distinct codes), not O(VINs)).

6. SEO Approach

Every decoded VIN gets a permanent, unique /vin/:vin URL with a generated meta description, Open Graph image, and Vehicle structured data. Option codes and colors double as their own evergreen pages (/learn/pccb, /learn/pts-colors) that rank independently of any single car.
